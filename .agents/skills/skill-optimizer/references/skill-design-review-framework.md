# Skill 设计评审框架

帮助 `skill-optimizer` 从"结构检查"升级为"设计质量评审"。

融合两类方法论：
1. Anthropic 实战经验：skill 是文件夹能力单元，description 是触发规则，善用渐进式披露/脚本/模板/hooks/config/memory
2. Google Cloud 5 种 Agent Skill 设计模式：Tool Wrapper / Generator / Reviewer / Inversion / Pipeline

---

## 一、判断目标 skill 的模式

先判断主模式，必要时标注"主模式 + 次模式"。

| 模式 | 核心价值 | 常配合 | 适合场景 | 高频问题 |
|------|----------|--------|----------|----------|
| **Tool Wrapper** | 提供按需知识 | `references/` | 文档知识库、SDK/API/CLI 用法 | description 太弱触发不到；知识堆在 SKILL.md 没拆 references；写了常识没写 gotchas |
| **Generator** | 稳定产出结构化结果 | `assets/` 模板 + `references/` 风格指南 | 文档/报告生成、模板化写作、结构化脚手架 | 输出结构不固定；模板该拆到 assets 却塞在 SKILL.md；风格规则和流程规则混在一起 |
| **Reviewer** | 评审、诊断、发现问题 | checklist / rubric | skill 审查、配置评审、代码 review、质量检查 | 只给结论没评价维度；审查标准分散不成体系；容易直接跳到修改绕过审查 |
| **Inversion** | 先提问/先收集信息，不直接产出 | 硬门槛 | 项目规划、需求澄清、高定制设计、复杂配置 | 应该先问却没问；缺少"信息不足时禁止继续"的硬门槛；一上来就输出大而全方案 |
| **Pipeline** | 按阶段顺序执行 | 检查点 + 阶段门槛 | 多步骤工作流、高风险操作流程 | 步骤顺序不清；缺少阶段门槛；未确认就进下一步；工作流过长没拆参考文件 |

---

## 二、判断模式是否匹配

识别完模式后继续判断：

- 当前模式是否适合这个 skill？
- 是否本该是 Generator 却写成了杂糅大说明书？
- 是否本该是 Inversion 却没有提问门槛？
- 是否本该是 Pipeline 却没有检查点？
- 是否是多个模式组合但主次关系不清？

输出时表达：
- **主模式：Pipeline；次模式：Generator**
- **问题：实际是 Pipeline，但没有显式阶段门槛，执行稳定性不足**

## 三、设计质量检查点

### 1. description 是否是触发规则
- 写清"做什么 + 什么时候用"
- 容易让模型判断是否应触发
- 写了不适用场景
- 不像人类简介，而是模型路由规则

### 2. 有没有 Gotchas
- 记录模型最容易犯的错
- 提醒误触发、误判、默认行为过强等问题

### 3. 是否善用文件夹能力
- 把 skill 当"文件夹能力单元"，不是单个 Markdown
- 合理使用 `references/` `scripts/` `assets/`
- 该拆不拆、全堆在 SKILL.md 的问题

### 4. 是否使用渐进式披露
- SKILL.md 只保留高层流程
- 细节拆到 references / templates / checklist
- 不一开始就把所有细节塞给模型

### 5. 是否写了显而易见的废话
- 重复模型本来就知道的常识
- 重点放在"容易做错什么"，不是"这是一个很厉害的 skill"

### 6. 是否缺 setup / config 思维
- 首次使用需要确认的配置
- 是否每次都让模型重复问同样的问题

### 7. 是否有 memory / logging 价值
- 适合沉淀历史记录的 skill
- 能否通过日志或状态文件提升稳定性

### 8. 是否该脚本化
- 高重复、可确定、适合脚本化的动作
- 该给 helper scripts 却还在用纯文字让模型手工做

### 9. 是否 railroading 或过度设计
- 指令写太死导致复用性变差
- 为简单场景引入过多结构

### 10. 是否考虑分发与复用
- 仅个人使用还是适合公开分发
- 公开分发时是否把私有路径/偏好/环境依赖写死了

## 四、审查结论中应补充的判断

除常规高/中/低优先级问题外，还建议补充：

**1. 模式判断**：主模式 + 次模式 + 当前问题描述

**2. 设计质量定位**：主要问题在哪一层——触发语义 / 工作流设计 / 结构拆分 / 边界确认门槛 / 缺少 Gotchas/scripts/templates/config

**3. 优先级排序**：
1. 先修触发失败、边界缺失、流程不可执行
2. 再修模式错位、结构臃肿、渐进式披露不足
3. 再修 Gotchas/scripts/assets/config 缺失
4. 最后做措辞、示例和可读性优化
---

## 五、常见误判提醒

- 不要为了套模式而套模式
- 不要一看到 skill 很长，就默认必须拆文件
- 不要把“用户只想微调 description”误做成整 skill 重构
- 不要因为 references 存在，就默认渐进式披露已经做得好
- 不要把“有流程”误认为“有门槛”
- 不要把“有脚本”误认为“设计就合理”

---

## 六、给 `skill-optimizer` 的使用建议

在审查目标 skill 时，建议流程如下：

1. 先读 `SKILL.md`
2. 判断主模式 / 次模式
3. 再决定要不要读 references / scripts / assets
4. 用本框架检查：
   - 触发语义
   - 模式匹配
   - 工作流门槛
   - 渐进式披露
   - Gotchas
   - 脚本化 / 模板化 / 配置化 / memory 化机会
5. 最后再输出优化计划，并等待用户确认
