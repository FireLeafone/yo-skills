# Skill 最佳实践（精炼版）

源：Anthropic 官方 skill best practices，仅保留对模型有增量价值的实操要点。

## 核心原则

- **默认假设模型已经聪明**：只写模型不知道的上下文。每条信息都要回答"模型真的需要这个吗？这段值得它的 token 成本吗？"
- **简洁是关键**：上下文窗口是共享资源，skill 的每个 token 都在和对话历史、其他 skill 元数据竞争。启动时只有 name + description 被预加载；SKILL.md 在触发时才加载，之后每个 token 都有成本。
- **自由度匹配任务脆弱性**：窄桥（低自由度）→ 具体护栏+精确步骤；开阔地（高自由度）→ 一般方向，信任模型选择路径。中间地带用伪代码或带参数的脚本。

## frontmatter 规则

- `name`：≤64 字符，小写字母/数字/连字符，无 XML 标签，无保留字。推荐动名词形式（`processing-pdfs`）或名词短语（`pdf-processing`）。避免模糊名（`helper`、`utils`）。
- `description`：≤1024 字符，非空，无 XML 标签。**用第三人称**写，因为 description 被注入系统提示用于触发路由。必须同时说明"做什么"和"什么时候用"，包含用户真实会说的关键词。避免"我可以帮助您…"这种第一人称或"帮助处理各种内容"这种空泛描述。

## 渐进式披露

SKILL.md 作为概述，指向模型按需查看的详细材料——像入职指南的目录，不是全部内容。

- **SKILL.md 正文 ≤500 行**；接近此限时拆到单独文件
- **所有 reference 文件直接从 SKILL.md 链接**，保持一级深度。避免深层嵌套引用（模型可能只 `head -100` 而漏掉信息）
- **长 reference（>100 行）顶部加目录**，让模型即使部分读取也能看到信息全貌
- **避免 SKILL.md 和 reference 重复大段内容**：SKILL.md 只写"什么时候读哪个 reference"

目录组织原则：
- 按域或功能组织（`reference/finance.md`、`reference/sales.md`），不要用无意义名（`docs/file1.md`）
- `assets/` 只放输出资源，不放说明文档
- 不要创建不必要的 README、CHANGELOG 等噪音文件

## 工作流与反馈循环

- **复杂任务用工作流**：分解为清晰的顺序步骤，提供可复制的 checklist 跟踪进度
- **实现反馈循环**：运行验证器 → 修复错误 → 重复。验证循环可以及早捕获错误
- **条件工作流**：通过决策点指导模型；如果工作流很大，推到单独文件并让模型按需读取

## 内容指南

- **避免时间敏感信息**：会过时的数据不要写进 skill；如果需要历史背景，用"旧模式"部分
- **术语一致**：选定一个术语，全 skill 统一使用（如始终"API 端点"，不要混用"URL"/"路由"/"路径"）
- **不要呈现多种方法**：除非必要，只给一种清晰路径，避免模型选择困难

## 脚本与可执行代码

- **脚本应解决问题，不推给模型**：处理错误条件，不要写"如果失败请检查配置"。配置参数应文档化，避免"巫毒常数"
- **预制脚本优于生成代码**：更可靠、省 token、省时间、保证一致性
- **明确执行意图**："运行 `analyze_form.py` 来提取字段"（执行）vs "参阅 `analyze_form.py` 了解算法"（作为参考读取）
- **创建可验证的中间输出**：对批量操作、破坏性更改、高风险操作，用"计划-验证-执行"模式。先输出 `changes.json`，用脚本验证后再执行
- **打包依赖**：在 SKILL.md 列出所需包，验证在执行环境可用
- **路径一律正斜杠**：`scripts/helper.py`，不用 `scripts\helper.py`
- **MCP 工具用全限定名**：`ServerName:tool_name`（如 `BigQuery:bigquery_schema`），避免"找不到工具"
- **不假设工具已安装**：不要写"使用 ffmpeg…"却不检查是否可用

## 运行时机制

模型访问 skill 的实际流程：
1. **元数据预加载**：启动时所有 skill 的 name + description 加载到系统提示
2. **按需读文件**：模型用 bash 读工具访问 SKILL.md 和其他文件
3. **高效执行脚本**：脚本通过 bash 执行，只有输出消耗 token
4. **大文件无上下文惩罚**：reference 在实际读取前不消耗 token

## 有效 Skill 检查清单

分享前验证：

**核心质量**
- description 具体且包含关键词 + 使用时机
- SKILL.md ≤500 行
- 详情在单独文件（如需要）
- 无时间敏感信息（或在"旧模式"部分）
- 术语一致
- 文件引用一级深
- 适当使用渐进式披露
- 工作流有清晰步骤

**代码和脚本**
- 脚本解决问题不推诿
- 错误处理明确有帮助
- 无巫毒常数
- 所需包已列出并验证可用
- 路径全用正斜杠
- 关键操作有验证步骤
- 质量关键任务有反馈循环
