# 请求代码审查

派遣代码审查子代理，在问题蔓延前及时发现。审查者会收到精心设计的评估上下文——绝不会是你的会话历史。这样能让审查者专注于工作成果，而非你的思考过程，同时保留你自己的上下文以便继续工作。

**核心原则：** 尽早审查，频繁审查。

## 何时请求审查

**必须执行：**
- 子代理驱动开发中的每个任务完成后
- 完成主要功能后
- 合并到 main 分支前

**可选但有价值：**
- 遇到瓶颈时（换个视角）
- 重构前（基线检查）
- 修复复杂 bug 后

## 如何请求

**1. 获取 git SHA：**
```bash
BASE_SHA=$(git rev-parse HEAD~1)  # 或 origin/main
HEAD_SHA=$(git rev-parse HEAD)

# 如果未 git add 的改动代码
git diff HEAD
```

**2. 派遣代码审查子代理：**

使用 `general-purpose` 类型的 Agent 工具，填写 `code-reviewer.md` 中的模板

**占位符：**
- `{DESCRIPTION}` - 所构建内容的简要摘要
- `{PLAN_OR_REQUIREMENTS}` - 应该做什么。如果存在设计文档、PRD 或需求文件，应在此明确引用路径，审查者会据此检查实现是否符合预期目标
- `{BASE_SHA}` - 起始提交
- `{HEAD_SHA}` - 结束提交

**3. 处理反馈：**
- 立即修复**严重**问题
- 在继续之前修复**重要**问题
- 记录**轻微**问题稍后处理
- 如果审查者错误，提出反驳（附理由）

## 示例

```
[刚完成任务 2：添加验证函数]

你：让我在继续之前请求代码审查。

BASE_SHA=$(git log --oneline | grep "Task 1" | head -1 | awk '{print $1}')
HEAD_SHA=$(git rev-parse HEAD)

[派遣代码审查子代理]
  DESCRIPTION: 添加了 verifyIndex() 和 repairIndex()，支持 4 种问题类型
  PLAN_OR_REQUIREMENTS: docs/yo/plans/deployment-plan.md 中的任务 2
  BASE_SHA: a7981ec
  HEAD_SHA: 3df7661

[子代理返回]：
  优点：架构清晰，测试真实
  问题：
    重要：缺少进度指示器
    轻微：魔法数字 (100) 用于报告间隔
  评估：可以继续

你：[修复进度指示器]
[继续任务 3]
```

## 与工作流集成

**子代理驱动开发：**
- 每个任务完成后审查
- 在问题累积前及时发现
- 在进入下一个任务前修复

**执行计划：**
- 每个任务完成后或自然检查点处审查
- 获取反馈，应用，继续

**临时开发：**
- 合并前审查
- 遇到困难时审查

## 危险信号

**绝对不要：**
- 因为"很简单"就跳过审查
- 忽视**严重**问题
- 带着未修复的**重要**问题继续
- 对有效的技术反馈进行争辩

**如果审查者错了：**
- 用技术理由提出反驳
- 展示代码/测试证明它是正确的
- 请求澄清

## Gotchas

- **未提交的改动无法通过 SHA 范围审查。** 如果有尚未 `git add` / `git commit` 的改动，先使用 `git diff HEAD` 查看差异并自行检查，或先提交后再派遣审查子代理。
- **Base SHA 选择错误会扩大或缩小审查范围。** 通常选择任务开始前的提交（`HEAD~1`）或 `origin/main`；若不确定，先用 `git log --oneline` 确认起点。
- **审查者可能误报。** 如果反馈与需求或设计冲突，先用代码/测试/文档说明反驳；若反馈合理但属于文档本身过时，应同时更新实现与相关文档。
- **引用设计文档/需求文件能显著提升审查质量。** 如果 `{PLAN_OR_REQUIREMENTS}` 只写一句摘要，审查者只能做代码质量检查，难以判断“是否满足需求目标”。
