## 通用规则

### 1、编码前先思考

不要假设。不要掩盖困惑。坦诚权衡。

实现之前：

- 明确陈述你的假设。如果不确定，就问。
- 如果存在多种理解，都要呈现出来——不要悄悄选一种。
- 如果有更简单的方案，说出来。必要时可以反对。
- 如果不清楚，停下来。指出哪里令人困惑。提问。

### 2、简单优先

最少代码解决问题。不要搞投机。

- 不要加超出需求的功能。
- 不要为一次性代码搞抽象。
- 不要搞"灵活"或"可配置"但没人要的东西。
- 不要处理不可能发生的错误场景。
- 如果写了 200 行但 50 行能搞定，重写。

问自己："资深工程师会说这过度复杂吗？"如果是，简化。

### 3、精准修改

只碰必须碰的。只清理自己弄的乱。

编辑现有代码时：

不要"优化"相邻的代码、注释或格式。
不要重构没坏的东西。
匹配现有风格，即使你自己会写成别的样子。
如果发现无关的死代码，提一下——不要删。
当你的改动产生了孤儿时：

删掉 YOUR 改动导致不再使用的导入/变量/函数。
不要删掉原先就存在的死代码，除非用户要求。
检验标准：每一行改动都应能追溯到用户的请求。

### 4、目标驱动执行

定义成功标准。循环验证直到通过。

把任务转化为可验证的目标：

"加验证" → "先写非法输入的测试，再让它们通过"
"修 bug" → "先写能复现 bug 的测试，再让它通过"
"重构 X" → "重构前后确保测试都通过"
多步骤任务，简要列出计划：

1. [步骤] → 验证：[检查]
2. [步骤] → 验证：[检查]
3. [步骤] → 验证：[检查]

明确的成功标准让你能独立推进。模糊的标准（"让它能跑"）需要不断确认。

### 5. cli command rules

- 在创建工作树或新分支之前，必须让我确认
- 不要主动执行 `git add/commit` 命令。我希望在审核之后再决定是否执行添加/提交操作。
