# 旅行策划全流程

## 0. 工具可用性

外部数据工具均为 **CLI（经 Bash 调用）**，不是 MCP 工具：

| 工具 | 调用 | 用途 |
|------|------|------|
| `flyai` | `flyai search-flight / search-hotels / search-poi / fliggy-fast-search / keyword-search` | 飞猪机票、酒店、门票实时数据与配图 |
| TikHub API | `python3 scripts/tikhub_xhs_search.py -k "关键词" [--fetch-detail] [--pretty]` | 小红书笔记搜索、正文与配图 |
| 网络搜索 | WebSearch / `mcp__grok-search__web_search` | 通用检索（降级与补充） |

用 Agent 并行搜索时，subagent prompt 中必须明确要求经 Bash 调 `flyai` 与 `scripts/tikhub_xhs_search.py`，仅在 CLI 失败时降级网络搜索。

## 1. 第一步：收集旅行要素

| 要素 | 必需 | 说明 |
|------|------|------|
| 出发地 | 是 | 决定交通方案与可达性 |
| 出行日期 | 是 | 具体日期或月份 |
| 天数 | 是 | 行程总天数 |
| 人员构成 | 是 | 有无老人小孩（含年龄） |
| 目的地 | 否 | 可空 → 进入目的地推荐流程 |
| 预算范围 | 否 | 总预算或每日预算 |
| 偏好 | 否 | 饮食禁忌、兴趣、小众/热门、体力 |

## 2. 第二步：目的地推荐（目的地未知时）

阶段间有依赖，**严禁跳过**：Phase A（候选短名单）→ Phase B（交通可行性）+ Phase B2（天气，可与 B 并行）→ Phase C（对比决策）。

### Phase A：候选发现与初筛（可按区域并行 Agent）

- **A1 候选发现**：用户给区域不给城市时，必须先搜索，严禁凭常识预设。
  - `flyai fliggy-fast-search --query "[区域] [天数]天 自由行"` —— 从真实旅行产品反推目的地；
  - `python3 scripts/tikhub_xhs_search.py -k "[区域] [月份] 旅游 推荐" --limit 8 --fetch-detail`；
  - 网络搜索攻略型推荐（`site:xiaohongshu.com` 等）。
- **A2 季节陷阱检查**：瀑布枯水期、花海/红叶花期、海滩台风/禁渔期、雪景化雪等——核心吸引物在出行时段是否最佳。
- **A3 特殊人群适配**：婴幼儿（推车友好度、母婴设施、医疗距离、安全风险、午休条件）；老人（高反、无障碍、医疗可及性）。

输出：候选短名单（每区域 3–5 个，跨区域去重，淘汰季节陷阱严重者）。

### Phase B：交通可行性（仅对短名单）

单程耗时判断：≤4h 理想；4–6h 可接受（占半天）；6–8h 勉强（带婴幼儿/老人谨慎）；>8h 除非行程 ≥7 天否则不推荐。查 12306 高铁、直飞航班、自驾距离，可并行 Agent。

### Phase B2：天气预报（出发 ≤15 天时强制执行）

数据源：中国天气网 15 天预报 → 和风天气 → AccuWeather。提取出行日每日天气状况、最高/最低气温、是否降雨。
淘汰规则：出行日 **≥50% 天数降雨** 标「天气风险高」降权；带婴幼儿/老人连续降雨 ≥3 天建议排除。

### Phase C：对比决策

横向对比表：交通可达性（权重最高）、出行日天气（最高）、特殊人群适配（高）、季节时令（高）、小众/人流（中）、景点丰富度（中）、费用（低）。给出明确排名与推荐理由，交用户决策。

## 3. 第三步：多维信息采集（目的地确定后，并行 Agent）

1. **精确天气**：Phase B2 已查则直接复用；否则按上面数据源补逐日预报 + 穿衣建议。近期出行可用 `curl "https://wttr.in/[城市]?format=j1"`。
2. **交通方案 + 衔接验证**：机票优先 `flyai search-flight --origin … --destination … --dep-date YYYY-MM-DD`（`adultPrice` 实时可标实查）。每个「A 交通 → B 交通」节点必须验证：到达时间 vs 末班时间、转场方式与耗时、缓冲 ≥1.5 小时；不可行则调整（更早航班/中途住一晚/机场包车），并在行程中标注衔接风险。
3. **景点与活动**：门票优先 `flyai search-poi --city-name … [--poi-level 5]`（`ticketInfo.price`、`freePoiStatus`）；攻略与小众推荐用搜索补充。
4. **美食餐饮**：必吃清单、人均、当地人推荐。
5. **住宿**：信息严格按 **携程 → 飞猪 → 通用搜索（标注参考）→ 人工**；报价按日期对齐后定实查/参考。
6. **小红书真实反馈**：每个核心景点（3–5 个）单独搜一次 `-k "[景点名] 实拍 打卡" --fetch-detail`，提取真实行程、避雷、带娃 tips 与图片 URL（取高赞 3–5 篇，每槽累计 4 张备选）。

**非酒店图片采集顺序（强制，前序命中即停）**：小红书（TikHub）→ 通用图库 → 兜底 → 人工素材。

## 4. 第四步：行程编排

原则：地理聚类（相近景点同一天）、节奏交替（暴走日后接休闲日）、就近用餐、每天 1–2 小时弹性、到达日只安排入住+周边。

特殊人群：
- 婴幼儿（0–3 岁）：每日最多 2 景点（上下午各 1），13:00–15:00 午睡段不排活动或排车程，标注推车/背带需求，餐厅选有儿童座椅的。
- 老人：每日步行 <1.5 万步，避免海拔 >3000m，安排午休。

每日结构参考：上午主景点 → 午餐（餐厅+特色菜+人均）→ 下午次景点/休闲 → 晚餐 → 夜间活动。

## 5. 第五步：预算编制

**每项费用必须标注可靠度**：实查（无标注，如 FlyAI 实时价、12306）/ 参考（`~`，搜索区间价）/ 估算（`≈`，经验推断）。

| 分类 | 注意 |
|------|------|
| 交通 | 大交通 + 市内；婴儿机票约成人 10%、婴儿高铁免票 |
| 住宿 | 房价 × 间数 × 晚数；4+ 大人通常 2 间房，多标参考价 |
| 餐饮 | 人均 × 人数 × 餐数；婴幼儿不单独计 |
| 门票 | 1.2m 以下 / 6 岁以下通常免票 |
| 其他 | 保险、伴手礼、杂项 |

预算区底部与 tips 必须含声明：「机票和酒店为动态定价，以上为参考估算。出行前请在携程/12306 确认实际价格并预订。」最终给出预估总费用 + 人均费用，并注明「含参考价成分，实际以预订为准」。

## 6. 第六步：生成 HTML

**两步法**：① 组装 tripData JSON（含 `generationDate`）写入 `[目的地]-[出发年月]/tripData.json`；② 渲染 HTML。

首选动态 UI（布局必须因目的地而异，不是换皮）：

| 维度 | 要求 | 示例 |
|------|------|------|
| 页面布局 | 必须不同 | Bento Grid / 杂志分栏 / 横向滚动 / 卡片瀑布流 |
| 行程展示 | 必须不同 | 纵向时间轴 / 横向日卡 / 手风琴 / 轮播 |
| 预算展示 | 必须不同 | 甜甜圈图 / 堆叠条形 / 环形进度 |
| 色调字体 | 必须不同 | 按目的地自然特征选配色 |
| 装饰元素 | 建议不同 | 目的地 motif（铁轨虚线、海浪、山水轮廓） |

技术红线：
1. 禁止深层嵌套 JS 模板字符串（复杂拼接用 `+` 或独立函数）；
2. SVG 图标必须有尺寸约束（外层 `width/height` 容器或自身属性）；
3. 特殊字符用 Unicode 转义（`·`→`\u00b7`、`°`→`\u00b0`、`¥`→`\u00a5`、`→`→`\u2192`）；
4. 生成后用 `node -e` 提取 `<script>` 做 `new Function()` 语法校验再交付。
5. 单文件 HTML（CSS/JS 全内联），离线可用，响应式 + 打印友好。

Fallback 固定模板：

```bash
python assets/generate.py tripData.json "目的地-N天旅行计划.html" \
  --auto-images --image-registry "assets/image_registry.sample.json" \
  --min-images 3 --ensure-day-images --min-images-per-activity 2 --save-updated-json
# 严格准确模式（禁用随机图兜底，宁缺毋滥）：加 --strict-images
```

**tripData 核心结构**：`title/dateRange/travelers/weather{summary,avgHigh,avgLow,rainfall,clothing,tips}/days[{date,weekday,theme,weather{icon,high,low},activities[{time,name,duration,cost,transport,note,imageUrl 或 meal{...}}]}]/hotels[{name,area,pricePerNight,highlights,rating,imageUrl}]/budget{transport,accommodation,food,tickets,other,total,perPerson}/tips[]`。`weather.icon` 可选：`sunny, cloudy, overcast, rainy, stormy, snowy, partlyCloudy`。完整样例见 `assets/roadbook-schema.sample.json`。

## 7. 第七步：部署与诚实声明

- Cloudflare Pages：新文件夹 + 更新 `index.html` 卡片链接（链接去掉 `.html` 后缀，中文目录名直接用）后 `git add/commit/push`，自动部署。构建配置：Build command 留空、输出目录 `/`、Deploy command `exit 0`。
- 交付时向用户说明数据可靠度：哪些是实查（天气、12306、FlyAI 实时价）、哪些是参考（搜索区间价）、哪些需用户确认（高铁衔接班次、出行前锁价预订）。

## 8. 附：行程详解文案规范（亮点 / 每日详述底稿）

适用：独立 Markdown 行程详解，或 roadbook `highlights`、`daily` 正文底稿。角色：**资深定制旅行顾问兼行程编辑**——松弛、好读、有度假感，不夸大、不标题党、不胡编景点。

**`# 行程亮点`**（唯一一级标题之一）：
- 至少覆盖游玩体验、住宿特色、美食亮点，可增设交通便利/文化体验/性价比与留白中的 1–2 项；
- 3–5 条无序列表，每条 1–2 句，合计约 200–300 字；
- 不写空洞口号，不写与报价矛盾的「全含」「赠送」。

**`# 每日行程详述`**：
- 二级标题 `## Day n：主题简述`（如 `## Day 1：抵达 XX，老城慢行`）；
- 从核心看点/值得体验/动线建议/停留时长量级/实用贴士中择 3–5 个重点，每日 ≤300 字；
- 以无序列表为主，关键信息加粗（**预约**、**省力走法**、**闭馆日**）。

质量标准：准确（不捏造时刻表/票价/房态）、实用（可指导当日节奏）、逻辑顺（动线时段不冲突）、有增量但不越素材边界、分层好扫读。资料不足处用「若…可…」「常见做法是…」区分推断与已确认事项；票价/开放时间不确定标 **待确认**。
