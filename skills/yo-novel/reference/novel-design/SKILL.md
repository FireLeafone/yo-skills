# novel-design

在小说正式动笔前，与用户一起完成前期设计规划，输出结构清晰、逻辑自洽、可指导后续写作的设计文档。

## 输出位置

- 默认输出到项目根目录下的 `presets/` 目录，与 `novel-create` 创建的项目结构保持一致。
- 如果 `presets/` 已存在同名文件，**必须先询问用户是覆盖、合并还是跳过**，禁止直接覆盖。
- 如果用户指定了其他目录，按用户要求执行。

## 设计文档清单

生成以下一个或多个文档，文件名与模板对应关系如下：

| 文档 | 输出文件 | 模板 |
|------|----------|------|
| 故事大纲 | `presets/outline.md` | `references/outline-template.md` |
| 人物设定 | `presets/characters.md` | `references/characters-template.md` |
| 世界观设定 | `presets/world.md` | `references/world-template.md` |
| 核心主题 | `presets/themes.md` | `references/themes-template.md` |
| 其他设定 | `presets/settings.md` | `references/settings-template.md` |

生成文档时，优先使用对应模板的结构，再根据用户需求填充内容。类型化指导参考 `references/novel-type.md`。

## 工作流程

### 1. 理解需求

- 识别用户想设计的部分：大纲、人物、世界观、主题、其他设定。
- 识别小说类型（如果没有，询问用户）。
- 识别用户已有的灵感或约束（主角名字、关键设定、结局倾向等）。

### 2. 确认设计计划（硬性门槛，但允许快速路径）

**常规路径：**
向用户展示设计计划，包含：
- 计划生成的文档清单
- 每个文档会覆盖的主要内容
- 文件存放路径

等待用户明确回复（如"确认"、"开始"、"ok"）后再继续。

**快速路径：**
如果用户已经明确指定了单一文档（例如"只写人物设定"），且提供了足够信息，可以直接生成该文档，但仍需先告知用户将创建哪个文件，并等待确认。

设计计划示例：

```markdown
计划为这本都市言情小说生成以下设计文档，存放在 `presets/`：
- `presets/characters.md` — 女主（律师）、男主（咖啡店老板）的角色卡与关系网
- `presets/outline.md` — 情感线大纲，从重逢到和解的关键节点
- `presets/themes.md` — 核心主题与情感基调

确认后我开始生成。
```

### 3. 头脑风暴与讨论

- 不要一次性输出完整文档后结束。
- 对关键设计选择，先给出 2-3 个可选方案或方向，说明利弊，邀请用户确认。
- 当用户提出修改时，立即更新对应部分，并检查是否与其他部分冲突。

### 4. 生成设计文档

- 按确认的计划使用模板结构生成文档。
- 内容完整、逻辑清晰、设定合理。
- 保持类型一致性：参考 `references/novel-type.md`。

### 5. 自洽性检查

- 世界规则是否自相矛盾。
- 人物动机是否与目标、背景一致。
- 大纲节点是否依赖尚未建立的设定。
- 如发现冲突，向用户指出并给出修改建议。

### 6. 报告结果

- 列出已生成的文件及路径。
- 简要说明每个文件的内容重点。
- 提示用户下一步可以开始基于这些设计写正文。

## 构建原则

1. **内部一致性**：规则不能自相矛盾。
2. **服务故事**：世界为故事服务，不要过度构建。
3. **展示而非讲述**：通过情节展现世界，不要只写说明文。
4. **留有空间**：给读者和后续创作保留想象空间。

## Gotchas

- **禁止直接覆盖已有文件**：`presets/` 下若已有同名文件，必须先询问用户。
- **不要把正文写进设计文档**：本 skill 只输出 `presets/` 下的设计文档，章节正文应放入 `chapters/`。
- **不要过度构建世界观**：世界设定要服务故事，避免在背景上投入过多而主线薄弱。
- **信息不足时不要直接输出完整方案**：如果用户只给了一句话灵感，先进入头脑风暴，而不是直接落盘 5 个完整文档。
- **保持跨文档一致**：修改人物设定后，要回头检查大纲和世界规则是否还成立。
