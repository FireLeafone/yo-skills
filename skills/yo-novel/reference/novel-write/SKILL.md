# novel-write

根据小说已有的大纲、人物、世界观和进度文档，撰写或续写单章正文，并保存为 Markdown 文件。

## When to Use

- 用户说"写第 X 章"、"帮我写一章"、"续写小说"、"开始写正文"
- 用户提到"novel chapter"、"write chapter"、"continue novel"、"next chapter"
- 用户已创建小说项目，需要基于 `presets/` 中的设计文档产出 `chapters/` 下的正文

When NOT to use:

- 用户还没有小说项目，想从零开始搭建结构（使用 `novel-create`）
- 用户只想做前期设计、大纲、人物设定，不需要落盘正文（使用 `novel-design`）
- 用户只是闲聊点子、询问写作技巧，没有明确要写某一章

## 输出位置

- 默认写入项目根目录下的 `chapters/` 目录，文件名为 `NNN-[章节名].md` 或用户指定的文件名。
- 如果用户明确指定了其他路径或文件名，按用户要求执行。
- 如果目标文件已存在，**必须先询问用户是覆盖、合并还是跳过**，禁止直接覆盖。

## 硬性规范

- **格式**：Markdown。
- **篇幅**：每章不少于 **2000** 字，不多于 **4000** 字。
- **章节状态**：写完后更新 `progress.md`，将该章标记为"已完成"或对应状态。
- **不使用占位符**：章节内容必须完整成文，不能使用"（此处省略）"、"待补充"等占位符。

## 工作流程

### 1. 定位目标章节

- 从用户输入中提取目标章节（如"第三章"、"chapter-003.md"）。
- 如果用户未明确指定，查看 `progress.md` 中标记为"待写"的下一章，并询问用户确认。

### 2. 读取项目文档

按以下顺序读取项目根目录下的文档：

1. `presets/outline.md` — 故事大纲，锁定目标章节的细纲与爽点安排。
2. `presets/characters.md` — 登场人物及其状态。
3. `presets/world.md` / `presets/settings.md` — 世界观与其他设定。
4. `progress.md` — 确认目标章节状态，回顾上一章结尾以保证衔接。
5. `rules.md` / `AGENTS.md` — 项目特定的写作规范。

如果缺少关键文档，先告知用户缺失内容，并询问是否继续或补充。

### 3. 记忆回溯

- 如果目标章节不是第一章，读取 `chapters/` 下最近 1-3 章的内容。
- 记录关键剧情、伏笔、人物状态、未解决的冲突，确保本章承接自然，避免"吃书"。

### 4. 确认内容方向（硬性门槛，但允许快速路径）

**常规路径：**
向用户展示本章计划，包含：

- 目标章节号与文件名
- 本章在大纲中的位置与核心任务
- 计划登场的主要人物
- 本章的关键冲突/爽点/转折点
- 拟采用的写作风格（参考下方"风格选择"）
- 文件保存路径

等待用户明确回复（如"确认"、"开始"、"ok"）后再继续。

**快速路径：**
如果用户已经明确提供了本章情节、风格、字数等要求，可以跳过详细计划，但仍需先告知用户将写入的文件路径，并等待确认。

### 5. 生成章节模板

仅在以下情况生成章节模板：

- 用户明确要求"先出大纲/框架"；或
- 本章场景超过 3 个、人物关系复杂、或多线交织；或
- 用户对本章情节只有模糊想法，需要通过模板对齐方向。

模板内容：

- 场景清单：时间、地点、人物、冲突、变化
- 情绪曲线：起 → 承 → 转 → 合
- 关键对话或伏笔提示

其他情况直接开始写正文，避免拖慢流程。生成模板前也要先告知用户并等待确认。

### 6. 撰写章节正文

按确认的内容方向撰写正文。核心要求：

- 叙事视角在本章内保持一致。
- 场景包含时间、地点、人物、冲突、变化。
- 对话推动情节、展现性格、有潜台词。
- 多用动作、表情、细节描写，减少直接心理说明。
- 节奏张弛有度，适当添加冲突、悬念、转折。
- 情节逻辑自洽，不违背已有设定。
- 埋设或回收伏笔，服务后续剧情。
- 按起承转合组织：起（引入）→ 承（推进）→ 转（转折）→ 合（落点/钩子）。

详细技巧参考 `references/writing-techniques.md`。

### 7. 风格选择

优先使用用户指定的风格。如果用户未指定，从以下内置风格中选择一种最贴合本章氛围的：

幽默风趣、犀利紧凑、叙事陈述、感人煽情、场面宏大、自然人声、古风文言、网文爽感。

选择依据：本章的情绪基调、场景规模、人物关系以及小说类型。不要一次性列出所有风格让用户挑选，而是给出你的推荐并说明理由；如果用户有明确偏好，按用户要求执行。

详细风格定义参考 `references/style-guide.md`。

### 8. 中断续写

- 检查当前项目是否有未完成的章节（`progress.md` 中标记为"写作中"或文件内容明显未完）。
- 如果存在未完成项，从断点继续创作，承接上文情节和氛围，保持人物语气与叙事节奏一致。
- 续写前先向用户说明将续写的文件与断点位置，等待确认。

### 9. Review 检查清单

写完后逐条自查：

- [ ] 字数在 2000-4000 之间。
- [ ] 没有出现"（待补充）"、"（略）"等占位符。
- [ ] 目标章节在大纲中的任务已完成。
- [ ] 登场人物的性格、动机与 `characters.md` 一致。
- [ ] 没有与 `world.md` / `settings.md` 矛盾的设定。
- [ ] 与上一章的结尾衔接自然。
- [ ] 关键伏笔已埋下或回收，且逻辑通顺。
- [ ] 对话、动作、表情描写足够，没有大量直接陈述。
- [ ] 章节末尾有适当的情绪落点或下一章钩子。

如发现问题，先修正后再保存。

### 10. 保存与报告

- 将正文写入 `chapters/` 下指定文件。
- 更新 `progress.md` 中该章节的状态。
- 向用户报告：文件路径、实际字数、本章核心事件、下一章建议或待填伏笔。

## Common Mistakes

- 没有读取 `presets/outline.md` 和 `progress.md` 就直接写 → 必须先了解本章任务与上下文。
- 不读取上一章内容就续写 → 会导致剧情断裂或人物状态错误。
- 字数不足 2000 或超过 4000 → 写完后检查字数。优先使用项目内的 `count_words.py`；如果不存在，统计中文字符数（CJK 字符）是否在 2000-4000 之间。
- 直接覆盖已有章节文件 → 必须先询问用户。
- 把正文写入 `presets/` → 正文应放入 `chapters/`。
- 风格不一致 → 同一章节内保持统一风格，切换视角或节奏时要有明确过渡。
- 过度使用直接陈述 → 多通过对话、动作、细节描写来展现信息。
