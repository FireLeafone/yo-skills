## 产品需求文档 (PRD)

### 概述

设计全面、生产级的产品需求文档 (PRD)，弥合业务愿景与技术实现之间的差距。这项技能适用于现代软件系统，确保需求得到清晰定义。

### 何时使用

以下情况适用：

- 启动新的产品或功能开发周期
- 将模糊的想法转化为具体的技术规范
- 定义人工智能驱动功能的需求
- 利益相关者需要一个统一的项目范围“权威来源”
- 用户要求“编写 PRD”、“记录需求”或“规划功能”

### 操作流程

#### 阶段 1：探索（访谈）

在编写 PRD 之前，您必须询问用户以填补知识空白。不要想当然地假设上下文。

**询问以下内容**：

- 核心问题：我们为什么要现在构建这个功能？
- 成功指标：我们如何知道它有效？
- 限制条件：预算、技术栈或截止日期？

### 阶段2：分析与范围界定

根据当前的对话上下文和代码库理解，综合用户反馈。识别依赖关系和潜在复杂性。

- 绘制用户流程图。
- 定义非目标以确保项目按时完成。

#### 阶段3：技术文档编写

使用下方严格的产品需求文档 (PRD) 结构生成文档。
如果没指定文档保存位置，则保存到 `docs/yo/prds/YYYY-MM-DD-<topic>-design.md`。

### PRD 质量标准

#### 需求质量

使用具体、可衡量的标准。避免使用“快速”、“简单”或“直观”等词语。

```markdown
# 模糊（不推荐）

- 搜索速度应快，并返回相关结果。
- 用户界面必须外观现代且易于使用。

# 具体（推荐）

+ 对于包含 1 万条记录的数据集，搜索必须在 200 毫秒内返回结果。
+ 搜索算法在基准测试中必须达到 85% 以上的精确度（精度@10）。
+ 用户界面必须遵循 Vercel/Next.js 设计系统，并达到 Lighthouse 无障碍访问评分 100%。
```

#### 严格的产品需求文档 (PRD) 结构

您必须严格按照以下结构编写输出：

1. 执行摘要

- 问题陈述：1-2 句话描述痛点。
- 解决方案：1-2 句话描述解决方案。
- 成功标准：3-5 个可衡量的关键绩效指标 (KPI)。

2. 用户体验与功能

- 用户画像：目标用户是谁？
- 用户故事：作为一名用户，我希望执行 [操作] 以获得 [收益]。
- 验收标准：每个用户故事的“完成”定义列表（要点式）。
- 非目标：我们不构建哪些功能？

3. 人工智能系统需求（如适用）

- 工具需求：需要哪些工具和 API？
- 评估策略：如何衡量输出质量和准确性。

4. 技术规范

- 架构概述：数据流和组件交互。
- 集成点：API、数据库和身份验证。
- 安全与隐私：数据处理和合规性。

5. 风险与路线图

- 分阶段发布：MVP -> v1.1 -> v2.0
- 技术风险：延迟、成本或依赖项故障。

### 实施指南

**务必**：

- 定义测试：对于人工智能系统，明确如何测试和验证输出质量。
- 迭代：提交草稿并就特定部分征求反馈意见。

**避免**：

- 跳过探索阶段：在编写产品需求文档 (PRD) 之前，至少要先提出两个澄清问题。
- 人为设定约束：如果用户没有指定技术栈，请询问或将其标记为待定 (TBD)。

### 示例：智能搜索系统

1. 概要

问题：用户难以在庞大的文档库中找到特定的文档片段。解决方案：一个智能搜索系统，能够提供带有来源引用的直接答案。成功：

- 搜索时间缩短 50%。\
- 引用准确率 >= 95%。

2. 用户故事

故事：作为一名开发者，我希望能够提出自然语言问题，这样我就不用猜测关键词了。
功能：

- 支持多轮提问。
- 返回带有“复制”按钮的代码块。

3. AI 系统架构

所需工具：codesearch、grep、webfetch。

4. 评估

- 基准测试：使用 50 个常见的开发者问题进行测试。
- 通过率：90% 的答案必须与预期引用匹配。
