# Yo-UI 技能索引

Yo-UI 是多个 UI/UX 设计辅助技能的集合入口。根据你的需求，自动路由到最合适的子技能。

## 可用子技能

**可以使用 /yo-ui help 列出下面所有子技能**

| 子技能 | 路径 | 用途 | 典型触发语 |
|--------|------|------|-----------|
| **layout** | `reference/layout/` | 修复布局结构问题：间距、层级、网格单调 | "布局不对"、"间距太乱"、"层级不清晰"、"卡片太单调" |
| **color** | `reference/color/` | 用战略性调色板替代灰度或单一强调色 | "配色太单调"、"颜色没层次"、"想加点颜色"、"品牌色怎么用" |
| **animate** | `reference/animate/` | 添加传达状态和反馈的动画 | "加动画"、"交互太生硬"、"缺少反馈"、"动效怎么做" |
| **adapt** | `reference/adapt/` | 将设计适配到不同屏幕/设备/平台 | "适配移动端"、"响应式怎么做"、"平板怎么显示"、"打印样式" |
| **optimize** | `reference/optimize/` | 识别并修复界面性能瓶颈 | "页面太慢"、"加载慢"、"卡顿"、"优化性能"、"LCP/CLS 太高" |
| **polish** | `reference/polish/` | 最终打磨：对齐、间距、状态、文案、边界情况 | "再打磨一下"、"细节不对"、"最终检查"、"上线前 review" |
| **critique** | `reference/critique/` | 专业设计评审，生成评估报告 | "评审设计"、"设计 review"、"看看哪里不对"、"设计健康度" |

**UI 设计改进流程**（建议按此顺序使用）：
1. **layout** → 修复空间结构和层级（地基）
2. **color** → 引入战略性颜色（视觉层次）
3. **animate** → 添加有意义的动效（交互反馈）
4. **adapt** → 适配多设备/平台（响应式）
5. **optimize** → 性能优化（加载和运行时）
6. **polish** → 最终打磨和审查（细节）

**独立评估**（不依赖上述流程）：
- **critique** → 设计评审、健康度评估、发现问题

## 路由规则

### 1. 显式指定（优先）

如果用户输入包含 `/yo-ui <子技能名>`，直接加载对应子技能：

- `/yo-ui layout ...` → 加载 `reference/layout/SKILL.md`
- `/yo-ui color ...` → 加载 `reference/color/SKILL.md`
- `/yo-ui animate ...` → 加载 `reference/animate/SKILL.md`
- `/yo-ui adapt ...` → 加载 `reference/adapt/SKILL.md`
- `/yo-ui optimize ...` → 加载 `reference/optimize/SKILL.md`
- `/yo-ui polish ...` → 加载 `reference/polish/SKILL.md`
- `/yo-ui critique ...` → 加载 `reference/critique/SKILL.md`

将 `/yo-ui <子技能名>` 之后的剩余内容作为任务传递。

### 2. 自动推断

如果用户只输入 `/yo-ui` 或 `/yo-ui <问题>` 但没有指定子技能名，根据用户意图推断：

| 用户意图关键词 | 匹配子技能 |
|--------------|-----------|
| 布局、间距、对齐、层级、网格、卡片单调、结构 | **layout** |
| 颜色、配色、调色板、品牌色、强调色、灰度、色彩 | **color** |
| 动画、动效、过渡、反馈、交互生硬、微交互 | **animate** |
| 响应式、移动端适配、平板、打印、多设备、屏幕适配 | **adapt** |
| 性能、慢、卡顿、加载、LCP、CLS、INP、优化 | **optimize** |
| 打磨、润色、细节、review、检查、上线前、最终审查 | **polish** |
| 评审、review、设计评估、健康度、哪里不对、 critique | **critique** |

**推断步骤：**
1. 分析用户问题的核心意图
2. 对照上表匹配最合适的子技能
3. 告知用户将使用哪个子技能，然后加载执行

### 3. 歧义处理

当用户意图同时涉及多个子技能（如"布局太乱而且颜色也不对"）：
1. 向用户说明各子技能的分工
2. 询问用户希望先执行哪个，或建议执行顺序（通常建议按 layout → color → ... 的顺序）
3. 按用户选择依次调用

## 工作流程

1. **解析用户输入**：检查是否包含 `/yo-ui <子技能名>` 的显式指定
2. **匹配子技能**：
   - 显式指定 → 直接使用对应子技能
   - 未指定 → 根据意图关键词自动推断
3. **加载并执行**：读取对应子技能的 SKILL.md，按其子技能的指令完成任务
4. **结果汇总**：将子技能的执行结果返回给用户

## 注意事项

- 本技能本身不执行具体任务，只负责路由到正确的子技能
- 子技能在 `reference/` 目录下
- 子技能的详细工作流程和规则，请参考各自的 SKILL.md
- **polish** 子技能会检查设计系统对齐，运行前确保设计系统文档可用
- **critique** 子技能会生成持久化快照，存储在 `docs/yo-ui/critique/` 目录下
