---

## 评估适配挑战

理解需要适配什么以及为什么：

1. **识别源语境**：
   - 最初是为谁设计的？（桌面网页？移动应用？）
   - 做出了哪些假设？（大屏幕？鼠标输入？快速网络？）
   - 在当前语境中哪些做得好？

2. **理解目标语境**：
   - **设备**：手机、平板、桌面、电视、手表、印刷？
   - **输入方式**：触摸、鼠标、键盘、语音、游戏手柄？
   - **屏幕限制**：尺寸、分辨率、方向？
   - **网络**：快速 WiFi、慢速 3G、离线？
   - **使用场景**：移动中 vs 办公桌前，快速浏览 vs 专注阅读？
   - **用户预期**：用户在这个平台上期待什么？

3. **识别适配挑战**：
   - 什么放不下？（内容、导航、功能）
   - 什么无法正常工作？（触摸上的悬停状态、过小的触摸目标）
   - 什么不合适？（桌面模式用在移动端、移动端模式用在桌面端）

**关键**：适配是为新语境重新思考体验，不是缩放像素。

## 规划适配策略

创建适合语境的策略：

### 移动端适配（桌面 → 移动端）

**布局策略**：
- 单列而非多列
- 垂直堆叠而非并排
- 全宽组件而非固定宽度
- 底部导航而非顶部/侧边导航

**交互策略**：
- 触摸目标最小 44×44px（不依赖悬停）
- 滑动手势（列表、轮播）
- 底部弹窗而非下拉菜单
- 拇指优先设计（控制按钮在拇指可及范围内）
- 更大的触摸区域和更宽的间距

**内容策略**：
- 渐进式披露（不要一次性展示所有内容）
- 优先展示主要内容（次要内容放入标签页/折叠面板）
- 更简洁的文字
- 更大的文字（最小 16px）

**导航策略**：
- 汉堡菜单或底部导航
- 降低导航复杂度
- 粘性头部提供上下文
- 导航流中的返回按钮

### 平板适配（混合方案）

**布局策略**：
- 双列布局（非单列或三列）
- 侧边面板展示次要内容
- 主从视图（列表 + 详情）
- 根据方向自适应（竖屏 vs 横屏）

**交互策略**：
- 同时支持触摸和指针
- 触摸目标 44×44px 但比手机布局更密集
- 侧边导航抽屉
- 适当的多列表单

### 桌面适配（移动端 → 桌面端）

**布局策略**：
- 多列布局（利用水平空间）
- 侧边导航始终可见
- 多个信息面板同时展示
- 固定宽度加最大宽度限制（不要拉伸到 4K）

**交互策略**：
- 悬停状态展示额外信息
- 键盘快捷键
- 右键上下文菜单
- 拖放
- Shift/Cmd 多选

**内容策略**：
- 预先展示更多信息（减少渐进式披露）
- 多列数据表格
- 更丰富的可视化
- 更详细的描述

### 印刷适配（屏幕 → 印刷）

**布局策略**：
- 在逻辑点分页
- 移除导航、页脚、交互元素
- 黑白（或有限颜色）
- 适合装订的页边距

**内容策略**：
- 展开缩短的内容（展示完整 URL、隐藏部分）
- 添加页码、页眉、页脚
- 包含元数据（打印日期、页面标题）
- 将图表转换为印刷友好版本

### 邮件适配（网页 → 邮件）

**布局策略**：
- 窄宽度（最大 600px）
- 仅单列
- 内联 CSS（无外部样式表）
- 基于表格的布局（为了邮件客户端兼容性）

**交互策略**：
- 大而明显的 CTA（按钮而非文字链接）
- 无悬停状态（不可靠）
- 复杂交互深度链接到网页应用

## 实施适配

系统地应用更改：

### 响应式断点

选择合适的断点：
- 移动端：320px–767px
- 平板：768px–1023px
- 桌面端：1024px+
- 或内容驱动的断点（设计在哪里崩溃就在哪里加断点）

### 布局适配技术

- **CSS Grid/Flexbox**：自动重排布局
- **容器查询**：基于容器而非视口适配
- **`clamp()`**：在最小值和最大值之间流体缩放
- **媒体查询**：不同语境不同样式
- **Display 属性**：按语境显示/隐藏元素

### 触摸适配

- 增大触摸目标尺寸（最小 44×44px）
- 增加交互元素之间的间距
- 移除依赖悬停的交互
- 添加触摸反馈（涟漪、高亮）
- 考虑拇指区域（底部比顶部更易触及）

### 内容适配

- 谨慎使用 `display: none`（仍然会下载）
- 渐进增强（核心内容优先，大屏幕加载增强）
- 视口外内容的延迟加载
- 响应式图片（`srcset`、`picture` 元素）

### 导航适配

- 将复杂导航在移动端转换为汉堡菜单/抽屉
- 移动应用的底部导航栏
- 桌面端常驻侧边导航
- 小屏幕上使用面包屑提供上下文

**重要**：在真机上测试。DevTools 的设备模拟有帮助但不完美。

**绝不**：
- 在移动端隐藏核心功能（如果重要，就让它正常工作）
- 假设桌面 = 高性能设备（考虑无障碍、老旧机器）
- 在不同语境使用不同的信息架构（会造成困惑）
- 破坏用户对平台的预期（移动端用户预期移动端模式）
- 忘记手机/平板上的横屏方向
- 盲目使用通用断点（使用内容驱动的断点）
- 忽略桌面上的触摸（很多桌面设备支持触摸）

## 验证适配

跨语境彻底测试：

- **真机**：在真实手机、平板、桌面上测试
- **不同方向**：竖屏和横屏
- **不同浏览器**：Safari、Chrome、Firefox、Edge
- **不同系统**：iOS、Android、Windows、macOS
- **不同输入方式**：触摸、鼠标、键盘
- **边界情况**：超小屏幕（320px）、超大屏幕（4K）
- **慢速网络**：在限速网络下测试

当适配在每个语境中感觉自然时，交给 `/yo-ui polish` 做最终润色。

## 响应式参考

适配流程在一个地方获得完整的响应式参考。

### 响应式设计

#### 移动优先：正确书写

从移动端的基准样式开始，使用 `min-width` 查询逐步叠加复杂度。桌面优先（`max-width`）意味着移动端先加载不必要的样式。

#### 断点：内容驱动

不要追逐设备尺寸；让内容告诉你哪里需要断点。从窄开始，拉伸直到设计崩溃，在那里添加断点。三个断点通常足够（640、768、1024px）。使用 `clamp()` 实现无需断点的流体值。

#### 检测输入方式，不只是屏幕尺寸

**屏幕尺寸不能告诉你输入方式。** 带触摸屏的笔记本电脑、带键盘的平板。使用 pointer 和 hover 查询：

```css
/* 精细指针（鼠标、触控板） */
@media (pointer: fine) {
  .button { padding: 8px 16px; }
}

/* 粗略指针（触摸、手写笔） */
@media (pointer: coarse) {
  .button { padding: 12px 20px; }  /* 更大的触摸目标 */
}

/* 设备支持悬停 */
@media (hover: hover) {
  .card:hover { transform: translateY(-2px); }
}

/* 设备不支持悬停（触摸） */
@media (hover: none) {
  .card { /* 无悬停状态 - 使用 active 替代 */ }
}
```

**关键**：不要依赖悬停实现功能。触摸用户无法悬停。

#### 安全区域：处理刘海屏

现代手机有刘海、圆角和主页指示器。使用 `env()`：

```css
body {
  padding-top: env(safe-area-inset-top);
  padding-bottom: env(safe-area-inset-bottom);
  padding-left: env(safe-area-inset-left);
  padding-right: env(safe-area-inset-right);
}

/* 带降级方案 */
.footer {
  padding-bottom: max(1rem, env(safe-area-inset-bottom));
}
```

**在 meta 标签中启用 viewport-fit**：
```html
<meta name="viewport" content="width=device-width, initial-scale=1, viewport-fit=cover">
```

#### 响应式图片：正确实现

##### 使用宽度描述符的 srcset

```html
<img
  src="hero-800.jpg"
  srcset="
    hero-400.jpg 400w,
    hero-800.jpg 800w,
    hero-1200.jpg 1200w
  "
  sizes="(max-width: 768px) 100vw, 50vw"
  alt="Hero image"
>
```

**工作原理**：
- `srcset` 列出可用图片及其实际宽度（`w` 描述符）
- `sizes` 告诉浏览器图片将显示的宽度
- 浏览器基于视口宽度和设备像素比选择最佳文件

##### 使用 Picture 元素进行艺术指导

当你需要不同的裁剪/构图（不只是分辨率）：

```html
<picture>
  <source media="(min-width: 768px)" srcset="wide.jpg">
  <source media="(max-width: 767px)" srcset="tall.jpg">
  <img src="fallback.jpg" alt="...">
</picture>
```

#### 布局适配模式

**导航**：三个阶段：移动端使用汉堡菜单 + 抽屉，平板使用水平紧凑，桌面端使用带标签的完整导航。**表格**：使用 `display: block` 和 `data-label` 属性在移动端转换为卡片。**渐进式披露**：使用 `<details>/<summary>` 折叠移动端内容。

#### 测试：不要只信任 DevTools

DevTools 设备模拟对布局有用，但缺少：

- 真实触摸交互
- 真实 CPU/内存限制
- 网络延迟模式
- 字体渲染差异
- 浏览器工具栏/键盘出现

**至少测试**：一部真实 iPhone、一部真实 Android、如相关的平板。便宜的安卓手机能揭示你在模拟器上永远看不到的性能问题。

---

**避免**：桌面优先设计。设备检测而非特性检测。分离的移动端/桌面端代码库。忽略平板和横屏。假设所有移动设备都性能强劲。
