## Gotchas

- **评分要诚实**：4 分意味着真正出色。大多数真实界面得分 20-32；不要默认给 3 分。
- **先视觉检查，后评分**：如有浏览器自动化可用，必须视觉检查后再打分。禁止仅凭源代码就给分。
- **提问必须具体**：向用户提出的问题必须引用报告中的具体发现。禁止问"你的受众是谁"这类通用问题。
- **临时文件必清理**：无论写入是否成功，临时正文文件必须删除。
- **持久化失败不阻塞评审**：如果 slug 计算失败或目标模糊，跳过持久化但继续完成完整评审。

## 工作流

复制此清单并跟踪进度：

```text
- [ ] 1. 设置：解析目标路径，计算 slug，读取 ignore.md
- [ ] 2. 设计评审：读取源文件 + 视觉检查，按维度评估
- [ ] 3. 生成报告：输出结构化评审报告
- [ ] 4. 等待确认：询问用户是否保存快照，确认后再写入
- [ ] 5. 持久化：写入快照，读取趋势，追加趋势行
- [ ] 6. 询问用户：基于发现提出 2-4 个有针对性的问题
```

## 设置

1. **将目标解析为**具体的文件路径或 URL。当两者标识同一个表面时，优先选择源路径而非开发服务器 URL；端口会漂移，路径不会。
   - "首页" -> `site/pages/index.astro` 或 `index.html`
   - "设置弹窗" -> 主要组件文件
   - "这个页面" -> 当前 URL 或源文件
2. **计算 slug**：
   ```bash
   node ./scripts/critique-storage.mjs slug "<resolved-path-or-url>"
   ```
   保留它。如果命令以非零退出，跳过本次运行的持久化和趋势，但继续评审。
3. **读取项目根目录 `docs/yo-ui/critique/ignore.md`**（如果存在）。静默丢弃匹配的发现问题；它是评审消耗的唯一先验运行输入。

## 设计评审

读取相关源文件。如有浏览器自动化可用，必须先视觉检查实时页面，再按下方维度评分。禁止仅凭源代码打分。

像设计总监一样思考，按以下维度评估：

评估维度：
- **AI 垃圾**：有人会立即相信"这是 AI 做的"吗？
- **整体设计**：层级、信息架构、情感契合、可发现性、构图、排版、颜色、无障碍、状态、文案和边界情况。
- **认知负荷**：使用 [references/cognitive-load.md](references/cognitive-load.md) 中的清单评估；报告失败项和可见选项 >4 的决策点。
- **情感旅程**：峰终规则、情感低谷、高 stakes 时刻的安抚。
- **Nielsen 启发式**：使用 [references/heuristic-scoring.md](references/heuristic-scoring.md) 中的评分指南；所有 10 条启发式评分 0-4。

返回：AI 垃圾判决、启发式评分、认知负荷、情感旅程、2-3 个优势、3-5 个优先问题、角色红旗、次要观察和挑衅性问题。

## 生成综合评审报告

评估综合成一份报告。聊天回复是主要面向用户的交付物。在聊天中展示下方完整的结构化评审；不要将其替换为摘要和链接。

像设计总监一样构建你的反馈：

#### 设计健康评分

以表格形式呈现 Nielsen 的 10 条启发式评分：

| # | 启发式 | 评分 | 关键问题 |
|---|-----------|-------|-----------|
| 1 | 系统状态可见性 | ? | [具体发现，如果良好则为 "n/a"] |
| 2 | 系统与现实世界匹配 | ? | |
| 3 | 用户控制与自由 | ? | |
| 4 | 一致性与标准 | ? | |
| 5 | 错误预防 | ? | |
| 6 | 识别优于回忆 | ? | |
| 7 | 灵活性与效率 | ? | |
| 8 | 美学与极简设计 | ? | |
| 9 | 错误恢复 | ? | |
| 10 | 帮助与文档 | ? | |
| **总计** | | **??/40** | **【评分段】** |

评分要诚实。4 分意味着真正出色。大多数真实界面得分 20-32。

#### 反模式判决

**从这里开始。** 这看起来像 AI 生成的吗？

**LLM 评估**：你自己对 AI 垃圾信号的评估。涵盖整体审美感觉、布局雷同、通用构图、错失的个性机会。

#### 整体印象

简短的直觉反应：什么有效，什么无效，以及最大的单一机会。

#### 做得好的地方

高亮 2-3 件做得好的事情。具体说明为什么有效。

#### 优先问题

按重要性排序的 3-5 个最具影响力的设计问题。

每个问题标记 **P0-P3 严重度**（定义见 [references/heuristic-scoring.md](references/heuristic-scoring.md)）：
- **[P?] 问题**：清晰命名问题
- **为什么重要**：这如何伤害用户或破坏目标
- **修复建议**：具体怎么做（要具体、可执行）

#### 角色红旗
> *参考 [references/role-testing.md](references/role-testing.md) 中的角色选择表。*

自动选择与此应用界面类型最相关的 2-3 个角色。如果用户指定了，则生成 1-2 个项目特定角色。

针对每个选定的用户角色，逐一分析其主要操作，并列出发现的具体问题：

**Alex（高级用户）**：未检测到键盘快捷键。表单需要点击 8 次才能完成主要操作。强制弹出式引导。放弃风险高。

**Jordan（新手）**：侧边栏仅使用图标导航。错误信息中包含技术术语（例如"404 Not Found"）。没有可见的帮助信息。会在第二步放弃。

务必具体。请明确指出导致每个用户角色操作失败的具体元素和交互方式。不要使用通用的用户角色描述；请具体说明导致他们操作失败的原因。

#### 次要观察

关于值得处理的小问题的快速笔记。

#### 值得思考的问题

可能解锁更好解决方案的挑衅性问题：
- "如果主要操作更突出会怎样？"
- "这需要感觉这么复杂吗？"
- "自信的版本会是什么样子？"

**记住**：
- 直接。模糊的反馈浪费每个人的时间。
- 具体。"提交按钮"，不是"某些元素"。
- 说出什么错了以及为什么对用户重要。
- 给出具体建议。完全删除"考虑探索……"。
- 无情地优先。如果一切都重要，什么都不重要。
- 不要软化批评。开发者需要诚实的反馈来交付出色的设计。

## 持久化快照

报告输出后，先询问用户是否将本次评审保存为快照。用户确认后，再写入 `docs/yo-ui/critique/<YYYY-MM-DD>_<slug>.md`，以便后续参考，也让 `/yo-ui polish` 可以在不复制粘贴的情况下获取优先问题。

如果用户拒绝保存，或设置中的 slug 为空（模糊或根级别目标），跳过本步骤，直接进入「询问用户」。

1. **将正文写入临时文件**，以便通过管道传递给脚本。使用完整的评审报告（启发式表、反模式判决、优先问题、角色红旗、次要观察和问题），但停在"询问用户"部分之前。
2. **通过 `YO_UI_CRITIQUE_META`（JSON）传递结构化元数据**，然后运行写入命令：
   ```bash
   YO_UI_CRITIQUE_META='{"target":"<user phrasing>","total_score":<n>,"p0_count":<n>,"p1_count":<n>}' \
     node ./scripts/critique-storage.mjs write <slug> <body-file>
   ```
   脚本会打印它写入的绝对路径。
3. **写入尝试完成后删除临时正文文件**，无论写入成功与否。如果删除失败，在最终输出中简要提及 `临时文件清理失败：<原因>`，但不要阻塞评审。
4. **读取趋势**获取语境：
   ```bash
   node ./scripts/critique-storage.mjs trend <slug> 5
   ```
   这会返回最近 5 个 frontmatter 条目（包括刚写入的那个）的 JSON 数组。
5. **在可见输出后追加一行**，在报告之后、问题之前：

   > **`<slug>` 的趋势（最近 5 次运行）：24 → 28 → 32 → 29 → 32**
   > 写入了 `docs/yo-ui/critique/<YYYY-MM-DD>__<slug>.md`。

   如果这是该 slug 的首次运行，趋势只有一个分数；如此说明："该目标首次运行，尚无趋势。"

用户确认后才执行写入。不要向用户展示脚本的 JSON 输出；只有人类可读的趋势行和写入路径。这里的失败不应阻塞流程其余部分；打印错误并继续。

## 询问用户

**展示发现后**，基于实际发现使用有针对性的问题。询问用户问题进行澄清，这些答案将塑造行动计划。

根据发现的内容提出以下类型的问题（适应具体发现；不要问通用问题）：

1. **优先方向**：基于发现的问题，询问用户现在最关注哪个类别。例如："我发现了视觉层级、颜色使用和信息过载的问题。我们应该先处理哪个领域？" 提供前 2-3 个问题类别作为选项。
2. **设计意图**：如果评审发现了调性不匹配，询问是否是有意的。例如："界面感觉临床化和企业化。这是预期的调性，还是应该感觉更温暖/更大胆/更活泼？" 基于能修复发现问题的方向提供 2-3 个调性方向作为选项。
3. **范围**：询问用户想处理多少。例如："我发现了 N 个问题。想全部处理，还是聚焦前 3 个？" 提供范围选项如"仅前 3 个"、"所有问题"、"仅关键问题"。
4. **约束**（可选；仅在相关时询问）：如果发现涉及很多领域，询问是否有禁区。例如："是否有部分应该保持原样？"

**问题规则**：
- 每个问题必须引用报告中的具体发现。永远不要问通用的"你的受众是谁？"问题。
- 最多 2-4 个问题。尊重用户的时间。
- 提供具体选项，不是开放式提示。
