# 启发式评分指南

在 0–4 分范围内对 Nielsen 的 10 条可用性启发式每项评分。要诚实：4 分意味着真正出色，不是"足够好"。

## Nielsen 的 10 条启发式

### 1. 系统状态可见性

通过及时、适当的反馈让用户了解正在发生什么。

**检查**：
- 异步操作期间的加载指示器
- 用户操作的确认（保存、提交、删除）
- 多步骤流程的进度指示器
- 导航中的当前位置（面包屑、激活状态）
- 表单验证反馈（内联，不只是提交时）

**评分**：
| 评分 | 标准 |
|------|------|
| 0 | 无反馈；用户在猜测发生了什么 |
| 1 | 很少反馈；大多数操作没有可见响应 |
| 2 | 部分；传达了一些状态，仍有重大缺口 |
| 3 | 好；大多数操作给出清晰反馈，有小缺口 |
| 4 | 出色；每个操作都有确认，进度始终可见 |

### 2. 系统与现实世界匹配

说用户的语言。遵循现实世界的惯例。信息以自然、逻辑的顺序出现。

**检查**：
- 熟悉的术语（没有未解释的行话）
- 符合用户预期的逻辑信息顺序
- 可识别的图标和隐喻
- 目标受众的领域适当语言
- 自然阅读流（从左到右、从上到下优先）

**评分**：
| 评分 | 标准 |
|------|------|
| 0 | 纯技术术语，对用户陌生 |
| 1 | 大多令人困惑；需要领域专业知识才能导航 |
| 2 | 混合；一些简单语言，一些行话泄露 |
| 3 | 大多自然；偶尔术语需要语境 |
| 4 | 始终流畅地说用户的语言 |

### 3. 用户控制与自由

用户需要从不想要的状态清晰"紧急退出"，无需冗长对话。

**检查**：
- 撤销/重做功能
- 表单和弹窗上的取消按钮
- 清晰导航回到安全地带（首页、上一步）
- 清除过滤器、搜索、选择的简单方式
- 从冗长或多步骤流程中退出

**评分**：
| 评分 | 标准 |
|------|------|
| 0 | 用户被困；没有刷新无法退出 |
| 1 | 退出困难；必须找到隐藏路径才能逃脱 |
| 2 | 有些退出；主要流程有逃脱，边界情况没有 |
| 3 | 控制好；用户可以退出和撤销大多数操作 |
| 4 | 完全控制；撤销、取消、返回和逃脱无处不在 |

### 4. 一致性与标准

用户不应该想知道不同的词语、情境或操作是否意味着同一件事。

**检查**：
- 整个界面术语一致
- 相同操作在所有地方产生相同结果
- 遵循平台惯例（标准 UI 模式）
- 视觉一致（颜色、排版、间距、组件）
- 一致的交互模式（相同手势 = 相同行为）

**评分**：
| 评分 | 标准 |
|------|------|
| 0 | 到处不一致；感觉像不同的产品拼接在一起 |
| 1 | 很多不一致；相似的东西看起来/行为不同 |
| 2 | 部分一致；主要流程匹配，细节分叉 |
| 3 | 大多一致；偶尔偏差，没有令人困惑的 |
| 4 | 完全一致；有凝聚力的系统，可预测的行为 |

### 5. 错误预防

比好的错误消息更好的是首先防止问题的设计。

**检查**：
- 破坏性操作前的确认（删除、覆盖）
- 防止无效输入的约束（日期选择器、下拉菜单）
- 减少错误的智能默认值
- 防止误解的清晰标签
- 自动保存和草稿恢复

**评分**：
| 评分 | 标准 |
|------|------|
| 0 | 容易犯错；没有任何护栏 |
| 1 | 很少防护；一些输入被验证，大多数没有 |
| 2 | 部分预防；常见错误被抓住，边界情况漏掉 |
| 3 | 好的预防；主动阻止了大多数错误路径 |
| 4 | 出色；通过智能约束错误几乎不可能 |

### 6. 识别优于回忆

最小化记忆负荷。使对象、操作和选项可见或易于检索。

**检查**：
- 可见选项（不埋在隐藏菜单中）
- 需要时的语境帮助（工具提示、内联提示）
- 最近项和历史
- 自动完成和建议
- 图标上的标签（不是仅图标导航）

**评分**：
| 评分 | 标准 |
|------|------|
| 0 | 重度记忆；用户必须记住路径和命令 |
| 1 | 大多回忆；很多隐藏功能，很少可见线索 |
| 2 | 有些辅助；主要操作可见，次要功能隐藏 |
| 3 | 好的识别；大多数东西可发现，很少记忆需求 |
| 4 | 一切可发现；用户永远不需要记忆 |

### 7. 使用的灵活性与效率

新手不可见的加速器加速专家交互。

**检查**：
- 常见操作的键盘快捷键
- 可定制的界面元素
- 最近项和收藏
- 批量操作
- 不复杂基础功能的高级用户功能

**评分**：
| 评分 | 标准 |
|------|------|
| 0 | 单一路径；没有捷径或替代方案 |
| 1 | 有限灵活性；主要路径的替代方案很少 |
| 2 | 有些捷径；基本键盘支持，有限的批量操作 |
| 3 | 好的加速器；键盘导航，一些定制 |
| 4 | 高度灵活；多条路径、高级功能、可定制 |

### 8. 美学与极简设计

界面不应包含无关或很少需要的信息。每个元素都应服务于目的。

**检查**：
- 每一步只展示必要信息
- 引导注意力的清晰视觉层级
- 颜色和有意的强调
- 没有争夺注意力的装饰性杂乱
- 专注、整洁的布局

**评分**：
| 评分 | 标准 |
|------|------|
| 0 | 压倒性；所有东西同等地争夺注意力 |
| 1 | 杂乱；太多噪音，难找重要的东西 |
| 2 | 有些杂乱；主要内容清晰，周边嘈杂 |
| 3 | 大多干净；专注的设计，少量视觉噪音 |
| 4 | 完美极简；每个元素都值得它的像素 |

### 9. 帮助用户识别、诊断和从错误中恢复

错误消息应该使用简单语言，精确指出问题，并建设性地建议解决方案。

**检查**：
- 简单语言错误消息（用户不使用错误码）
- 具体问题识别（"Email 缺少 @" 而非 "无效输入"）
- 可操作的恢复建议
- 问题源附近显示错误
- 非阻塞错误处理（不要清空表单）

**评分**：
| 评分 | 标准 |
|------|------|
| 0 | 晦涩错误；代码、行话或根本没有消息 |
| 1 | 模糊错误；"出错了"没有指导 |
| 2 | 清晰但无用；命名问题但没有修复 |
| 3 | 清晰有建议；识别问题并提供下一步 |
| 4 | 完美恢复；精确定位问题、建议修复、保留用户工作 |

### 10. 帮助与文档

即使系统无需文档即可使用，帮助也应该容易找到、以任务为中心且简洁。

**检查**：
- 可搜索的帮助或文档
- 语境帮助（工具提示、内联提示、引导游览）
- 以任务为中心的组织（非功能组织）
- 简洁、可扫描的内容
- 不离开当前语境即可轻松访问

**评分**：
| 评分 | 标准 |
|------|------|
| 0 | 没有任何帮助 |
| 1 | 帮助存在但难找或无关 |
| 2 | 基本帮助；FAQ 或文档存在，非语境 |
| 3 | 好的文档；可搜索，大多以任务为中心 |
| 4 | 出色的语境帮助；在正确时刻提供正确信息 |

---

## 评分总结

**总分**：40 分（10 条启发式 × 最高 4 分）

| 分数范围 | 评级 | 含义 |
|----------|------|------|
| 36–40 | 出色 | 仅需轻微润色；可以发布 |
| 28–35 | 好 | 处理薄弱区域，基础扎实 |
| 20–27 | 可接受 | 用户满意前需要重大改进 |
| 12–19 | 差 | 需要重大 UX 改造；核心体验损坏 |
| 0–11 | 严重 | 需要重新设计；当前状态不可用 |

## 问题严重度 (P0–P3)

对评分期间发现的每个问题标记优先级：

| 优先级 | 名称 | 描述 | 操作 |
|--------|------|------|------|
| **P0** | 阻塞 | 完全阻止任务完成 | 立即修复；这是致命问题 |
| **P1** | 重大 | 造成重大困难或困惑 | 发布前修复 |
| **P2** | 轻微 | 令人烦恼，但有变通方案 | 下次迭代修复 |
| **P3** | 润色 | 最好修复，没有真实用户影响 | 有时间就修复 |

**提示**：如果不确定两个级别之间，问："用户会为此联系支持吗？"如果是，至少 P1。
