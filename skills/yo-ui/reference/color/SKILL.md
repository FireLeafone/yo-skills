用战略性调色板替代保守的灰度或单一强调色设计：选择配色策略，挑选适合品牌的色相家族，然后有意图地应用颜色。更多颜色 ≠ 更好。战略性配色胜过彩虹。

## 登记

品牌：调色板就是声音。首先根据 SKILL.md 选择配色策略（克制 / 投入 / 全色板 / 浸透），并遵循其用量。投入、全色板和浸透有意超过 ≤10% 规则；那条规则仅适用于克制。允许意外组合；当所选策略要求时，主色可以主导页面。

产品：语义优先，几乎总是克制。强调色保留给主要操作、当前选中和状态指示器。不是装饰。每个颜色在每个屏幕上都有统一的一致含义。

在选择颜色之前，先制定**色彩策略**。色彩策略可分为四个阶段：
- **克制**：使用带色调的中性色，并搭配一种占比≤10%的强调色。产品默认色；品牌极简主义。
- **饱和**：使用一种饱和度高的颜色，覆盖30-60%的面积。品牌默认色，适用于以品牌识别为导向的页面。
- **完整调色板**：3-4个命名角色，每个角色都经过精心设计。品牌推广活动；产品数据可视化。
- **饱和**：色彩覆盖整个表面。品牌主打色；推广活动页面。

## 评估色彩机会

分析当前状态并识别机会：

1. **理解当前状态**：
   - **缺少颜色**：纯灰度？有限的中性色？一个胆怯的强调色？
   - **错失机会**：哪里颜色可以增加含义、层级或愉悦？
   - **语境**：对这个领域和受众什么是合适的？
   - **品牌**：是否有应该使用的现有品牌色？

2. **识别颜色增加价值的地方**：
   - **语义含义**：成功（绿）、错误（红）、警告（黄/橙）、信息（蓝）
   - **层级**：吸引注意力到重要元素
   - **分类**：不同的部分、类型或状态
   - **情感基调**：温暖、活力、信任、创意
   - **导航**：帮助用户导航和理解结构
   - **愉悦**：视觉趣味和个性的时刻

如果代码库中有任何不清楚的地方，询问澄清。

**关键**：更多颜色 ≠ 更好。战略性配色每次都胜过彩虹 vomit。每个颜色都应该有目的。

## 规划色彩策略

创建有目的的色彩引入计划：

- **调色板**：哪些颜色匹配品牌/语境？（中性色之外最多选择 2-4 种颜色）
- **主色**：哪种颜色拥有 60% 的彩色元素？
- **强调色**：哪些颜色提供对比和高亮？（30% 和 10%）
- **应用策略**：每种颜色出现在哪里，为什么？

**重要**：颜色应该增强层级和含义，而非制造混乱。越重要的地方越要克制。

## 战略性引入颜色

系统地添加颜色，跨越以下维度：

### 语义色
- **状态指示器**：
  - 成功：绿色系（翡翠、森林、薄荷）
  - 错误：红/粉色系（玫瑰、深红、珊瑚）
  - 警告：橙/琥珀色系
  - 信息：蓝色系（天空、海洋、靛蓝）
  - 中性：用于非激活状态的灰/石板色

- **状态标签**：状态的彩色背景或边框（激活、待处理、已完成等）
- **进度指示器**：显示完成度或健康度的彩色条、环或图表

### 强调色应用
- **主要操作**：为最重要的按钮/CTA 上色
- **链接**：为可点击文本添加颜色（保持无障碍）
- **图标**：为关键图标着色以提升识别度和个性
- **标题/标签**：为章节标题或关键标签添加颜色
- **悬停状态**：交互时引入颜色

### 背景与表面
- **着色背景**：如果用品牌色替换纯灰色，朝品牌色相倾斜，而非朝通用暖冷对倾斜。默认暖色色调（`oklch(97% 0.01 60)` 及其邻近色）现在是 AI 奶油/沙色的标志。要针对品牌具体化或保持中性。
- **彩色区域**：使用微妙的背景色分隔区域
- **渐变背景**：用微妙、有意的渐变增加深度（不要通用紫蓝渐变）
- **卡片与表面**：将卡片或表面朝向品牌色倾斜，而非下意识"为了温暖"

**使用 OKLCH 处理颜色**：它在感知上是均匀的，意味着亮度的等步长*看起来*是相等的。非常适合生成和谐的色彩阶梯。

### 数据可视化
- **图表与图形**：使用颜色编码类别或值
- **热力图**：颜色强度显示密度或重要性
- **对比**：不同数据集或时间段的颜色编码

### 边框与强调
- **细线边框**：完整周边 1px 彩色边框（不是侧边条纹；绝对禁止 `border-left/right > 1px`）
- **下划线**：用于强调或激活状态的有色下划线
- **分隔线**：微妙的彩色分隔线替代灰色线条
- **焦点环**：匹配品牌的彩色焦点指示器
- **表面色调**：4-8% 强调色背景洗色替代条纹

**绝不**：`border-left` 或 `border-right` 大于 1px 作为彩色强调条纹。这是父 skill 中三个绝对禁令之一。如果想将卡片标记为"激活"或"警告"，使用完整细线边框、背景色调、前置符号或编号前缀。不是侧边条纹。

### 排版颜色
- **彩色标题**：对章节标题使用品牌色（保持对比度）
- **高亮文本**：用于强调或分类的颜色
- **标签与标签**：用于元数据或分类的小型彩色标签

### 装饰元素
- **插画**：添加彩色插画或图标
- **形状**：品牌色的几何形状作为背景元素
- **渐变**：彩色渐变叠加或网格背景
- **斑点/有机形状**：用于视觉趣味的柔和彩色形状

## 平衡与精修

确保颜色添加改善了体验而非压倒它：

### 保持层级
- **主色** (60%)：主要品牌色或最常用的强调色
- **次要色** (30%)：增加变化的辅助色
- **强调色** (10%)：关键时刻的高对比
- **中性色**（剩余）：结构用的灰/黑/白

### 无障碍
- **对比度**：确保 WCAG 合规（文本 4.5:1，UI 组件 3:1）
- **不单独依赖颜色**：在颜色之外使用图标、标签或图案
- **色盲测试**：验证红/绿组合对所有用户都有效

### 一致性
- **一致的调色板**：使用定义的调色板中的颜色，而非任意选择
- **系统化应用**：颜色含义始终一致（绿色始终 = 成功）
- **温度一致**：暖色板保持暖色，冷色板保持冷色

**绝不**：
- 使用彩虹中的每种颜色（中性色之外选择 2-4 种颜色）
- 无语义含义地随机应用颜色
- 在彩色背景上使用灰色文本。它看起来褪色；使用背景色的深色阴影或透明度替代
- 违反 WCAG 对比度要求
- 使用颜色作为唯一指示器（无障碍问题）
- 让所有东西都多彩（违背了目的）
- 默认使用紫蓝渐变（AI 垃圾审美）

## 验证颜色添加

测试色彩化改善了体验：

- **更好的层级**：颜色是否适当地引导注意力？
- **更清晰的含义**：颜色是否帮助用户理解状态/分类？
- **更吸引人**：界面是否感觉更温暖、更有吸引力？
- **仍然无障碍**：所有颜色组合是否都满足 WCAG 标准？
- **不过度**：颜色是否平衡且有目的？

当调色板证明了自己的价值时，交给 `/yo-ui polish` 做最终润色。

## 实时模式签名参数

当从实时模式调用时，每个变体必须声明一个 `color-amount` 参数，以便用户可以在克制强调色和浸透表面之间调节，而无需重新生成。针对 `var(--p-color-amount, 0.5)` 编写变体的 CSS，通常作为背景上的 alpha 乘数，或作为 OKLCH 表达式中色度轴的缩放因子。0 = 中性/单色，1 = 全饱和/主导覆盖。

```json
{"id":"color-amount","kind":"range","min":0,"max":1,"step":0.05,"default":0.5,"label":"Color amount"}
```

在之上叠加 1-2 个变体特定参数：调色板选择（带命名选项的 `steps`）、温度暖度，或色调 vs 真色。

## 色彩参考

以便色彩化流程在一个地方获得完整的色彩参考。

### 色彩与对比

#### 色彩空间：使用 OKLCH

**停止使用 HSL。** 改用 OKLCH（或 LCH）。它在感知上是均匀的，意味着亮度的等步长*看起来*是相等的，不像 HSL 中 50% 亮度的黄色看起来很亮而 50% 的蓝色看起来很暗。

OKLCH 函数接受三个分量：`oklch(lightness chroma hue)`，其中 lightness 是 0–100%，chroma 大致是 0–0.4，hue 是 0–360。要构建主色及其浅/深变体，保持 chroma+hue 大致不变并变化 lightness，但在接近白或黑时**降低 chroma**，因为极端亮度的高 chroma 看起来刺眼。

你选择的 hue 是品牌决策，不应该来自默认值。不要下意识选择蓝色（hue 250）或暖橙色（hue 60）；这些是最主要的 AI 设计默认值，不是任何特定品牌的正确答案。

#### 构建功能调色板

##### 着色中性色

**纯灰色已死。** 零 chroma 的中性色在彩色品牌旁边感觉没有生气。给所有中性色添加微小的 chroma 值（0.005–0.015），朝向品牌色相倾斜。chroma 小到不会在意识上被感知为"着色"，但它创建了品牌色和 UI 表面之间的潜意识凝聚力。

你朝向倾斜的 hue 应该来自**这个**项目的品牌，而不是"暖 = 友好，冷 = 科技"的公式。如果你的品牌色是蓝绿色，你的中性色就朝蓝绿色倾斜。如果你的品牌色是琥珀色，它们就朝琥珀色倾斜。关键是与**特定**品牌的凝聚力，而不是通用调色板。

**避免**总是朝暖橙色或总是朝冷蓝色倾斜的陷阱。那两个是最懒的默认值，它们在不同项目之间创造了自己的单一文化。

##### 调色板结构

完整系统需要：

| 角色 | 目的 | 示例 |
|------|---------|---------|
| **主色** | 品牌、CTA、关键操作 | 1 种颜色，3-5 个色阶 |
| **中性色** | 文本、背景、边框 | 9-11 个色阶 |
| **语义色** | 成功、错误、警告、信息 | 4 种颜色，每种 2-3 个色阶 |
| **表面色** | 卡片、模态框、遮罩 | 2-3 个层级 |

**除非需要，否则跳过次要/第三色。** 大多数应用使用一种强调色即可。添加更多会造成决策疲劳和视觉噪音。

##### 60-30-10 规则（正确应用）

这条规则是关于**视觉权重**，而非像素数量：

- **60%**：中性背景、留白、基础表面
- **30%**：次要色：文本、边框、非激活状态
- **10%**：强调：CTA、高亮、焦点状态

常见错误：到处使用强调色因为"它是品牌色"。强调色之所以有效*是因为*它们稀有。过度使用会扼杀它们的力量。

#### 对比度与无障碍

##### WCAG 要求

| 内容类型 | AA 最低 | AAA 目标 |
|--------------|------------|------------|
| 正文 | 4.5:1 | 7:1 |
| 大文本（18px+ 或 14px 粗体） | 3:1 | 4.5:1 |
| UI 组件、图标 | 3:1 | 4.5:1 |
| 非必要装饰 | 无 | 无 |

##### 危险的色彩组合

这些通常对比度失败或造成可读性问题：

- 白色上的浅灰色文本（#1 无障碍失败）
- 绿色背景上的红色文本（或相反）：8% 的男性无法区分这些
- 红色背景上的蓝色文本（视觉上振动）
- 白色上的黄色文本（几乎总是失败）
- 图片上的细浅色文本（对比度不可预测）

##### 测试

不要相信你的眼睛。使用工具：

- [WebAIM Contrast Checker](https://webaim.org/resources/contrastchecker/)
- 浏览器 DevTools → Rendering → 模拟视觉缺陷
- [Polypane](https://polypane.app/) 用于实时测试

#### 主题：浅色与深色模式

##### 深色模式不是浅色模式的反转

你不能只是交换颜色。深色模式需要不同的设计决策：

| 浅色模式 | 深色模式 |
|------------|-----------|
| 阴影表现深度 | 更亮的表面表现深度（无阴影） |
| 深色文本在浅色上 | 浅色文本在深色上（降低字重） |
| 鲜艳的强调色 | 略微降低强调色饱和度 |
| 白色背景 | 纯黑或适合品牌的深色表面（品牌着色的近黑色在 oklch 12–18% 也有效） |

在深色模式中，深度来自表面亮度而非阴影。构建一个 3 步表面阶梯，更高层级更亮（例如 15% / 20% / 25% 亮度）。使用与品牌色**相同**的 hue 和 chroma（无论这个项目是什么；不要下意识选择蓝色），只变化 lightness。略微降低正文文本字重（例如 350 替代 400），因为深色上的浅色文本比浅色上的深色文本看起来更重。

##### Token 层级

使用两层：原始 token（`--blue-500`）和语义 token（`--color-primary: var(--blue-500)`）。对于深色模式，只重新定义语义层；原始 token 保持不变。

#### Alpha 是设计异味

大量使用透明度（rgba、hsla）通常意味着不完整的调色板。Alpha 创建不可预测的对比度、性能开销和不一致性。为每种语境定义明确的叠加色替代。例外：焦点环和需要穿透的交互状态。
