检测器和自动化 QA 输出只是缺陷证据。干净的脚本结果从来不是设计强大的证明；收集浏览器证据并检查真实交互路径。

## 设计系统发现

将功能对齐到设计系统**不是可选的**。没有对齐的打磨是在漂移之上的装饰，它让下一个人的工作更难。发现先于任何其他打磨工作。

1. **找到设计系统**：搜索设计系统文档、组件库、样式指南或 token 定义。研究核心模式：设计原则、目标受众、颜色 token、间距刻度、排版样式、组件 API、动效约定。
2. **记录约定**：共享组件如何导入？使用什么间距刻度？哪些颜色来自 token vs 硬编码值？建立了什么动效和交互模式？可比较操作使用什么流程形状（弹窗 vs 全页、内联 vs 路由、失焦保存 vs 显式提交）？
3. **识别漂移，然后命名根本原因**：对每个偏差，分类为**缺失 token**（值应该存在于系统中但不存在）、**一次性实现**（共享组件已存在但未使用）或**概念性偏差**（功能的流程、信息架构或层级与相邻功能不匹配）。按类别修复：修补值、切换到共享组件、或重做流程。不命名原因就修复症状是漂移累积的方式。

如果存在设计系统，打磨**必须**将功能对齐到它。如果不存在，根据代码库中可见的约定打磨。**如果系统的任何内容含糊，询问。永远不要猜测设计系统原则。**

## 打磨前评估

触碰任何东西前理解当前状态和目标：

1. **审查完整性**：
   - 功能完整吗？
   - 有需要保留的已知问题吗？（用 TODO 标记）
   - 质量标准是什么？（MVP vs 旗舰功能？）
   - 何时发布？（有多少打磨时间？）

2. **以体验为先思考**：谁实际使用这个，对他们来说最好的体验是什么？有效设计胜过装饰性打磨；看起来漂亮但阻碍用户流程的功能不算打磨。打开 DevTools 前从他们的视角走过路径。

3. **识别打磨区域**：
   - 视觉不一致
   - 间距和对齐问题
   - 交互状态缺失
   - 文案不一致
   - 边界情况和错误状态
   - 加载和过渡流畅度
   - 信息架构和流程漂移（这个功能揭示的复杂度是否与相邻功能一致？）

4. **拉入任何先前评审**（可选信号）：如果 `/yo-ui critique` 已在同一目标上运行过，其优先问题是有用的先验，用于决定先处理什么。将目标解析为文件路径或 URL，然后：
   ```bash
   slug=$(node ./scripts/critique-storage.mjs slug "<resolved>")
   node ./scripts/critique-storage.mjs latest "$slug"
   ```
   以 body = 找到退出 0；将 P0/P1 项目折叠进你的打磨列表并提及快照路径，让用户看到你读了什么。以 2 退出 = 无快照，继续没有它。评审是众多输入之一。无论如何都要自己做一遍。

5. **分类美观 vs 功能**：将每个问题分类为**美观的**（看起来不对，不阻碍用户）或**功能的**（破坏、阻塞或困惑体验）。打磨时间紧时，功能问题先发布；美观的可以后续跟进。质量应该一致；永远不要完美一个角落而另一个粗糙。

**关键**：打磨是最后一步，不是第一步。不要打磨功能不完整的工作。

## 系统化打磨

系统地处理以下维度：

### 视觉对齐与间距

- **像素级对齐**：所有东西对齐到网格
- **一致间距**：所有 gap 使用间距刻度（没有随机的 13px gap）
- **光学对齐**：为视觉重量调整（图标可能需要偏移以光学居中）
- **响应式一致性**：间距和对齐在所有断点工作
- **网格遵守**：元素对齐基线网格

**检查**：
- 启用网格叠加并验证对齐
- 用浏览器检查器检查间距
- 在多个视口尺寸测试
- 寻找"感觉"不对的元素

### 信息架构与流程

视觉打磨在形状错误的流程上是浪费的工作。将体验的*形状*匹配到系统，不只是表面。

- **渐进式披露**：与相邻功能相比，匹配揭示多少、何时揭示。一个设置页面暴露 40 个字段而应用其余部分一次揭示 5 个是漂移，即使每个字段都完美样式化。
- **已建立的用户流程**：多步操作遵循与别处可比较流程相同的形状：弹窗 vs 全页、内联编辑 vs 单独路由、失焦保存 vs 显式提交、乐观 vs 悲观更新。
- **层级与复杂度**：相同的概念权重在整个过程中获得相同的视觉权重。主要操作不会在产品的一个角落变成第三级，第三级操作也不会大喊大叫。
- **空状态、加载和到达过渡**：内容如何到达、更新和离开与相邻功能一致。
- **命名和心理模型**：功能使用与系统其余部分相同的名词和动词。这里的"Workspace"不应该在三屏之外变成"Project"。

### 排版精修

- **层级一致性**：相同元素始终使用相同尺寸/字重
- **行长度**：正文 45-75 个字符
- **行高**：适合字体大小和语境
- **孤行与寡行**：最后一行没有单个词
- **断字**：适合语言和栏宽
- **字距**：在需要的地方调整字母间距（特别是标题）
- **字体加载**：没有 FOUT/FOIT 闪烁

### 颜色与对比

- **对比度**：所有文本满足 WCAG 标准
- **一致 token 使用**：没有硬编码颜色，全部使用设计 token
- **主题一致性**：在所有主题变体中工作
- **颜色含义**：相同的颜色始终表示相同的意思
- **无障碍焦点**：焦点指示器可见且对比度足够
- **彩色背景上的灰色**：永远不要在彩色背景上放灰色文本；使用该颜色的阴影或透明度

### 交互状态

每个交互元素需要所有状态：

- **默认**：静止状态
- **悬停**：微妙反馈（颜色、缩放、阴影）
- **焦点**：键盘焦点指示器（永远不要移除而不替换）
- **激活**：点击/触摸反馈
- **禁用**：明显不可交互
- **加载**：异步操作反馈
- **错误**：验证或错误状态
- **成功**：成功完成

**缺失的状态造成困惑和损坏的体验**。

### 微交互与过渡

- **平滑过渡**：所有状态变化适当动画化（150-300ms）
- **一致缓动**：使用 ease-out-quart/quint/expo 实现自然减速。永远不要弹跳或弹性；它们感觉过时。
- **无卡顿**：流畅动画；在增加润色时使用氛围 blur/filter/mask/shadow 效果，但限制昂贵的绘制区域并避免随意布局属性动画
- **适当动效**：动效服务目的，不是装饰
- **减少动效**：尊重 `prefers-reduced-motion`

### 内容与文案

- **一致术语**：相同的东西始终叫相同的名字
- **一致大小写**：标题大小写 vs 句首大小写一致应用
- **语法与拼写**：没有错别字
- **适当长度**：不太啰嗦，不太简短
- **标点一致性**：句子末尾有句号，标签没有（除非所有标签都有）

### 图标与图片

- **一致风格**：所有图标来自同一家族或匹配风格
- **适当尺寸**：图标在语境中尺寸一致
- **正确对齐**：图标在光学上与相邻文本对齐
- **Alt 文本**：所有图片都有描述性 alt 文本
- **加载状态**：图片不导致布局偏移，正确的宽高比
- **Retina 支持**：高 DPI 屏幕的 2x 资源

### 表单与输入

- **标签一致性**：所有输入正确标签
- **必填指示器**：清晰且一致
- **错误消息**：有用且一致
- **Tab 顺序**：逻辑键盘导航
- **自动聚焦**：适当（不要过度使用）
- **验证时机**：一致（失焦 vs 提交）

### 边界情况与错误状态

- **加载状态**：所有异步操作都有加载反馈
- **空状态**：有帮助的空状态，不只是空白
- **错误状态**：清晰的错误消息及恢复路径
- **成功状态**：成功操作的确认
- **长内容**：处理超长名称、描述等
- **无内容**：优雅处理缺失数据
- **离线**：适当的离线处理（如适用）

### 响应式

- **所有断点**：测试移动端、平板、桌面端
- **触摸目标**：触摸设备上最小 44×44px
- **可读文本**：移动端文字不小于 14px
- **无水平滚动**：内容适配视口
- **适当重排**：内容逻辑适配

### 性能

- **快速初始加载**：优化关键路径
- **无布局偏移**：元素加载后不跳动（CLS）
- **流畅交互**：无延迟或卡顿
- **优化图片**：适当的格式和尺寸
- **延迟加载**：视口外内容延迟加载

### 代码质量

- **移除 console.log**：生产环境没有调试日志
- **移除注释代码**：清理死代码
- **移除未使用导入**：清理未使用的依赖
- **一致命名**：变量和函数遵循约定
- **类型安全**：没有 TypeScript `any` 或忽略的错误
- **无障碍**：适当的 ARIA 标签和语义 HTML

## 打磨清单

系统地检查：

- [ ] 对齐到设计系统（漂移被命名并按根本原因解决）
- [ ] 信息架构和流程形状匹配相邻功能
- [ ] 所有断点视觉对齐完美
- [ ] 间距一致使用设计 token
- [ ] 排版层级一致
- [ ] 所有交互状态已实现
- [ ] 所有过渡流畅（60fps）
- [ ] 文案一致且打磨
- [ ] 图标一致且尺寸适当
- [ ] 所有表单正确标签和验证
- [ ] 错误状态有帮助
- [ ] 加载状态清晰
- [ ] 空状态友好
- [ ] 触摸目标最小 44×44px
- [ ] 对比度满足 WCAG AA
- [ ] 键盘导航工作
- [ ] 焦点指示器可见
- [ ] 没有控制台错误或警告
- [ ] 加载时无布局偏移
- [ ] 在所有支持的浏览器中工作
- [ ] 尊重减少动效偏好
- [ ] 代码干净（没有 TODO、console.logs、注释代码）

**重要**：打磨关乎细节。放大看。眯眼看。亲自使用它。小事累积。

关注细节。放大直到对齐正确且间距读出为有意为之。然后发布。

**绝不**：
- 功能不完整前打磨
- 不对齐设计系统就打磨；那是在漂移之上装饰
- 猜测设计系统原则而非在含糊时询问
- 如果 30 分钟后发布则花几小时打磨（分类）
- 打磨时引入 bug（彻底测试）
- 忽略系统性问题（如果到处间距都不对，修复系统，不只一个屏幕）
- 完美一件事而留下其他粗糙（一致的质量水平）
- 设计系统等价物存在时创建新的一次性组件
- 硬编码应该使用设计 token 的值
- 引入与已建立模式或流程分叉的新模式或流程

## 最终验证

标记为完成前：

- **亲自使用它**：实际与功能交互。
- **在真机上测试**：不只是浏览器 DevTools。
- **请他人评审**：新鲜的眼睛能发现东西。
- **与设计对比**：匹配预期设计。
- **检查所有状态**：不要只测试快乐路径。
- **谨慎对待自动化**：在可用且相关时运行检测器或 QA 命令，修复它们的缺陷，但永远不要引用干净结果作为工作已打磨的证明。

## 清理

打磨后确保代码质量：

- **替换自定义实现**：如果设计系统提供了你重新实现的组件，切换到共享版本。
- **移除孤立代码**：删除打磨后未使用的样式、组件或文件。
- **整合 token**：如果你引入了新值，检查它们是否应该成为 token。
- **验证 DRY**：查找打磨期间引入的重复并整合。
